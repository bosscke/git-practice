Git New Files

  • Tracked - files that Git knows about and are added to the repository
  • Untracked - files that are in your working directory, but not added to the repository
git status

Git Staging Environment

Staged files are files that are ready to be commited.

git add index.html

Using --all will stage all changes.

git add --all

Shorthand command is for add is . or -A.

git add -A

Git Commit

By adding clear messages to each commit, it is easy for you and others to see what has changed and when!

git commit -m "First release"

Check the status of the repo using --short option to see changes in a more compact way.

  • ?? - Untracked files
  • A - Files added to stage
  • M - Modified files
  • D - deleted files
git status --short

It is possible to commit changes directly, skipping the staging environment. The -a option will automatically stage every chnged already tracked file.

git commit -a -m "Update .. "

See the history of commits with log command.

git log

Git Branch

In Git, a branch is a new/separate version of the main repository. Branches allow you to work on different parts of a project without impacting the main branch.

git branch git-practice-2

To confirm that we created a new branch:

git branch

The * beside master indicates that we are currently still on the master branch. To move to the new created branch git-practice-2 use command checkout.

git checkout git-practice-2

To go back to master branch :

git checkout master

Create new branch to deal with a fix..:

git checkout -b emergency-fix

Git Branch Merge

Now we can merge the current branch (master) with emergency-fix. First make sure to be on master branch.

git checkout master

Now merge (master) and emergency-fix.

git merge emergency-fix

As master and emergency-fix are same, it can be deleted.

git branch -d emergency-fix

Git and GitHub

GitHub Get Started

Create new repository. And copy the URL.

git remote add origin URL

Check if add was successful.

git remote -v

Now push the master to the origin url, and set it as default remote branch.

git push --set-upstream origin master

Or short:

git push -u origin master

Pull from GitHub

When working as a team on a project, it is important to that everyone is up to date. Anytime you start working on a project, you should get the most recent changes to your local copy. In Git this is done with pull.

pull is a combination of 2 different commands:

  • fetch
  • merge

fetch gets all the changes of a tracked branch/repo, on your local Git and, fetch updates to see what has changed on GitHub.

git fetch origin

Git Hub Flow

The GitHub flow focuses on branching and makes it possible for teams to experiment and make deployments regularly. The GitHub flow works like this:

  1. Create a new branch
  2. Make changes and commits
  3. Open a pull request
  4. Review
  5. Deploy
  6. Merge

    7. Create a New Branch

    Branching is the key concept in Git, and it works around that the master branch is, always deployable. If you want to try something new you create a new branch, this gives you the chance to make changes without affecting the main branch.

    When your new branch is ready, it can be reviewed and discussed and merged with main branch if ready.

    Make Changes and Add Commits

    In the new branch meke changes by adding, editing and deleting files. Whenever you raeach a milestone add the changes to the branch by commit.

    Adding commits keeps track of your work. Each commit should have a message explaining what has changed and why. Each commit becomes a part of the history of the branch and a point you can revert back to if you need.

    Open a Pull Request

    Pull requests are key part of GitHub. A Pull request notifies people you have changes ready for them to consider or review.

    Ask others to review your changes or pull your contribution an merge it into their branch.

    Review

    Pull redquest can be review by whoever has the propper access to the branch. Git Hub shows new commit and feedback in the "unified Pull Request view"

    Deploy

    When the pull request has been reviewed and all looks good, it's time for testing. Git Hub allows to deploy from a branch for final testing in production before merging with the master branch.

    If any issues arise you can undo the the changes by deploying the master branch into production again!

    Merge

    Once the testing is finished merge the code into the master branch. Pull request keep records of changes to your code, and if you comment and name changes well you can go back and understand why changes and decisions were made.
